## 按严重度分组的问题清单

## P0-01 · Ghost 按钮在 sheet 内完全失能（全局 3 弹窗）

**现象**
退出弹窗（`quiz.confirm_exit`）、续答弹窗（`mid_quiz`）、结算更多 sheet（`result.cta.more`）、重玩确认（`dynasty.replay_confirm`）里的 `.btn--ghost` 按钮在移动端被渲染成"一行居中加粗灰字"，看不出是按钮：

- `bg: rgba(0,0,0,0)`（透明）
- `border: 0`（无边）
- `color: #7a6040`（muted，与正文对比度不足）
- `.dialog__actions flex-direction: column` + `.btn { inline-size:100% }` → 占满整宽
- 与 `.dialog__title` 视觉难区分（title 22px/semibold #1C0F00；ghost btn 16px/semibold #7a6040，居中 100% 宽）

**用户口述**："全局弹窗里的按钮乱七八糟没有主次统一" 就是指这个——他根本没把 ghost 识别成按钮。

**问题本质**：可供性（affordance）缺失。ghost 样式适合 hover 生效的桌面端 inline 次要操作，不适合移动端全宽 sheet 场景。

**修复方案**（style.css，同时修 3 处）

在 `.dialog__actions` 作用域内给 ghost 补上足以识别的视觉线索：

```css
/* 替换现有 style.css 约 1196-1203 行 的 .btn--ghost 规则 */
.btn--ghost {
  background: transparent;
  color: var(--color-fg-secondary);  /* 不用 muted，对比度上到 AA */
}
@media (hover: hover) and (pointer: fine) {
  .btn--ghost:hover { background: var(--color-bg-secondary); color: var(--color-fg-primary); }
}
.btn--ghost:active { opacity: 0.75; }

/* 在 dialog 里 ghost 必须自带边框作为 affordance */
.dialog__actions .btn--ghost {
  background: transparent;
  border: 1.5px solid var(--color-border-subtle);
  color: var(--color-fg-secondary);
}
```

**同时调整 app.js** — 下面 P0-02~P0-04 会重构弹窗按钮数量与 variant，ghost 在某些场景直接消失，但这个样式 fallback 必须先修到位，否则 result 页 sheet 仍出问题。

## P0-02 · 退出弹窗 3 按钮语义重叠（quiz.confirm_exit）

**现象**（`/quiz/xia` → 返回键）
- 「放弃这一关」ghost — 清 session（实际实现：850 行） + **未 navigate**（bug，见 P0-06）
- 「回首页（下次继续）」outline — 保留 session navigate home
- 「继续答题」primary — dismiss

用户原话：**"左边 2 个按钮是一个意思，不要重复"** — 虽然实现语义不同（清 vs 留），但文案未能传达差异：两者都写"回首页/这一关…"让人以为都是离开。

**问题本质**：语义去重 + 文案失焦。HIG/MD 二次确认弹窗推荐 2 选项（破坏性 + 取消），第三个选项=信息重复。

**修复方案**（app.js 1265-1282 行）

退出弹窗必须 2 按钮。保留进度是默认行为（符合用户心智），放弃需要二次确认或放到 menu。

```javascript
// app.js 1265-1282 整块替换
function openExitConfirm() {
  openDialog({
    role: 'alertdialog',
    title: t('quiz.confirm_exit.title'),       // "要走了吗？"
    body: t('quiz.confirm_exit.body'),         // 改为："你的进度会保留，下次回来接着答。"
    actions: [
      { label: t('quiz.confirm_exit.leave'), variant: 'outline', onClick: function () {
        // 保留 session → 回家
        navigate('/home');
      }},
      { label: t('quiz.confirm_exit.continue'), variant: 'primary', autofocus: true, onClick: function () {} }
    ]
  });
}
```

**同时改 index.html（zh + en）的 confirm_exit 节**（188-192 行 / 1342-1348 行）

```json
// zh
"confirm_exit": {

  "continue": "继续答题",
```

"放弃这一关"（destructive）挪到首页长按卡片菜单（future work），不放在退出路径里—— 退出时触发放弃是低频操作且用户心智危险。

## P0-03 · 续答弹窗 3 按钮语义重叠（mid_quiz）

**现象**（home 页加载 600ms 后自动弹）
- 「放弃，回首页」ghost — 清 session（不 navigate，因已在 home） + **未 rerender 首页**，卡片仍显示"继续（1/5）"
- 「从头再来」outline — 清 session 后 navigate quiz/xia → getQuizSession 重建 session
- 「接着答」primary — navigate quiz/xia → getQuizSession 命中已有 session

语义矩阵：

| 按钮 | 清进度 | 去哪 |
|---|---|---|
| 放弃 | ✓ | 留在 home |
| 从头再来 | ✓ | quiz（新 session） |
| 接着答 | ✗ | quiz（复用 session） |

其实用户心智只有 2 件事："要不要继续之前那局"：
- 是 → 接着答
- 否 → 放弃 / 从头

"放弃"和"从头再来"在用户看来都是"不要之前那局了"，没必要把"从头再来"在 home 入口独立 —— 用户如果点了"放弃"然后再点卡片，就会进入新 session，效果就是"从头再来"。

**问题本质**：过度选项（Hick's Law 违反）+ 状态未刷新。

**修复方案**（app.js 833-861 行）

改成 2 按钮，"discard" 走 dismiss（什么都不做=过会儿提示），"continue" 正面引导：

```javascript
// app.js 833-861 整块替换
function maybePromptMidQuiz(state) {
  var sess = state.currentSession;
  if (!sess || !sess.dynastyId) return;
  try {
    if (sessionStorage.getItem('skipMidQuizOnce')) {
      sessionStorage.removeItem('skipMidQuizOnce');
      return;
    }
  } catch (e) {}
  var dynName = t('dynasty.' + sess.dynastyId + '.name');
  openDialog({
    role: 'dialog',
    title: t('mid_quiz.title'),
    body: t('mid_quiz.body', { dynasty: dynName, current: sess.currentQuestionIdx + 1 }),
    actions: [
      { label: t('mid_quiz.cta.later'), variant: 'outline', onClick: function () {
        // 本次会话不再提示，但不清 state，下次 session 还会提示
        try { sessionStorage.setItem('skipMidQuizOnce', '1'); } catch (e) {}
      }},
      { label: t('mid_quiz.cta.continue'), variant: 'primary', autofocus: true, onClick: function () {
        navigate('/quiz/' + sess.dynastyId);
      }}
    ]
  });
}
```

**index.html** zh 1052-1060 / en 2207+

```json
// zh
"mid_quiz": {
  "title": "你有一关没走完",
  "body": "在 {dynasty}，停在了第 {current} 题。",
  "cta": {
    "continue": "接着答",
    "later": "先不了"
  }
}
```

"从头再来"功能保留：首页已通关卡片点击→ replay_confirm 负责，未通关不存在"从头"概念。

## P0-04 · 结算页 primary CTA 与真实重要操作错位

**现象**（`/result/xia`）
底部 sticky result-ctas：
- 左侧 primary `[进入 商 →]` 占主位（除非没有下一朝代）
- 右侧 ghost ellipsis `[…]` 方形 56×56 图标按钮，里面藏着：
  - ghost 「回到朝代图谱」
  - primary 「再玩一次这一关」

**3 个问题**：

1. **primary 重复**（P0）：result 页 primary 是"进入 商"，但点 ellipsis 后弹出的 sheet 又给了另一个 primary（再玩一次）。同一屏幕两个 primary 争夺用户注意，违反「每弹窗 ≤ 1 primary」规则。

2. **重要操作被藏**（P0）：用户刚答完夏朝 3/10，**真实意图极可能是"再答一次这一关"提分**，而不是强推下一朝代。当前把再玩藏进 ellipsis = 3 次点击（点 ellipsis → 识别菜单 → 点再玩）才能开始。

3. **ellipsis 菜单内标题空洞**：sheet 标题只有"更多"，没 body，两个选项一 ghost 一 primary，ghost 又是失能状态，菜单变成"只有一个按钮的 sheet"。

**问题本质**：菜单把"非主要操作"变成"隐藏的重要操作"；信息密度不足但层级错位。

**修复方案**（app.js 1366-1398 行）

取消 ellipsis 菜单，改成 3 颗可见按钮（primary 1 + outline 2），按用户意图优先级从高到低排列：

```javascript
// app.js 1366-1398 替换整块
'<div class="result-ctas-sticky">' +
  '<button type="button" class="btn btn--primary" data-replay>' + esc(t('result.cta.replay_this')) + '</button>' +
  (nextDyn
    ? '<button type="button" class="btn btn--outline" data-next-dyn>' + esc(t('result.cta.next_dynasty', { dynasty: t('dynasty.' + nextDyn + '.name') })) + ' ' + iconSvg('arrow-right') + '</button>'
    : '') +
  '<button type="button" class="btn btn--ghost" data-home>' + esc(t('result.cta.home')) + '</button>' +
'</div>' +
```

并把事件绑定（1380-1398 行）替换：

```javascript
var replayB = $('[data-replay]', app);
if (replayB) replayB.addEventListener('click', function () {
  var s = readState(); s.currentSession = null; s.lastResult = null; writeState(s);
```

**Perfect 结算例外**（9/10+ 或 10/10）：用户全对时应该推下一朝代，此时交换 primary：

## P0-06 · Discard 路径不 navigate 不 rerender → 死循环

**现象**
1. 首页弹 mid_quiz，点「放弃，回首页」→ session 被清，**但首页不重渲染**，卡片仍显示"继续（1/5）"。
2. 用户以为没生效，再点卡片 → `getQuizSession`（app.js 944-956）检测到 state.currentSession 已 null，新建 session 写回 state。
3. 用户觉得卡顿或返回 home → 600ms 后又弹 mid_quiz。
4. 循环。

**问题本质**：discard 的 onClick 只操作 state 不操作视图，违反"每次操作需有可见反馈（Nielsen #1）"。

**修复方案**

在 P0-03 的 mid_quiz 重构里，"later"不再清 state（用 sessionStorage 标记本次 session 不再提示）。若仍保留"放弃"按钮，必须 rerender：

```javascript
// 如果选择保留 discard 选项（见 P0-03 替代方案）
{ label: t('mid_quiz.cta.discard'), variant: 'ghost', onClick: function () {
  var s = readState(); s.currentSession = null; writeState(s);
  // rerender home
  var appEl = document.getElementById('app');
  if (appEl) renderHome(appEl);
}},
```

另外 **app.js 845（mid_quiz）、1271（confirm_exit）的 discard 分支，目前都没 navigate**。quiz 页的 discard 必须 `navigate('/home')`，否则用户放弃后还在 quiz 页看着题目。

app.js 1271-1274 行替换：

```javascript
{ label: t('quiz.confirm_exit.discard'), variant: 'ghost', onClick: function () {
  var s = readState(); s.currentSession = null; writeState(s);
  navigate('/home');
}},
```

但按 P0-02 方案，confirm_exit 直接删掉 discard 按钮，此问题也就不存在了。

```javascript
function onDynastyClick(dynId, st) {
```

## P1-09 · 弹窗按钮顺序不统一（有的 ghost 在最下，有的在最上）

**现象**
- mid_quiz: `[ghost discard, outline restart, primary continue]` — 顺序 destructive → neutral → primary
- confirm_exit: `[ghost discard, outline keep, primary continue]` — 同上
- replay_confirm: `[ghost cancel, primary ok]` — cancel 居然也用 ghost
- result.more sheet: `[ghost home, primary replay]` — 同上

**各弹窗 ghost 语义含糊**：有时是 destructive（discard），有时是 cancel（先不了），有时是 neutral（home）。

**问题本质**：variant 语义不统一，下游无法靠 variant 识别操作类型。

**修复方案**：统一约定（写入 CLAUDE.md / button-system-spec.md）：

| 操作类型 | variant | 位置（column 布局时 top→bottom） |
|---|---|---|
| destructive（清数据） | danger 或 outline + 危险色文字 | 最上或独立行 |
| cancel / dismiss | outline | primary 上一行 |
| primary confirm | primary | 最下 |
| neutral navigation（次要跳转） | ghost | 最下方独立区 |

按此约定，上面 4 个弹窗按 P0-02~P0-04 重构后就自然统一成 `[outline, primary]` 二按钮，ghost 在"4 dialog" 场景全部消失（只在 result sticky CTAs 中作为"回首页"的弱引导保留）。

## P1-12 · 首页"继续（1/5）"数字含义不统一

**现象**
截图里首页夏朝卡显示"继续（1/5）"但题目总数是 10。进度状态 in_progress 按道理应该是"当前题号/总题数"，但 5 从哪来？看 loadQuizMeta 是按 count 字段算的，实际上某些朝代题库可能只有 5 题。用户困惑：到底 1/5 还是 1/10？

**问题本质**：数字来源不透明。

**修复方案**：把"1/5"改成"接着第 2 题"或"已答 1 道"（单值，避免分数歧义）。index.html zh:

```json
"dynasty.status.in_progress": "接着第 {current} 题"
```

## P1-13 · 结算页"看看刚才哪里差一点"入口与 ctas 分离

**现象**
结算页中间有 `result-wrong-entry` 大卡片（"看看刚才哪里差一点"+右箭头，344×75 全宽），而底部 sticky 又有 primary + ghost。这两组构成"3 层行动"：wrong-review → primary → ghost sheet。用户眼球路径：上→下→sheet，层级太深。

**问题本质**：结算页有太多并列 CTA。

```css
/* style.css */
.result-wrong-entry {

  padding: 12px 0;
  min-block-size: 44px;
  font-size: 15px;
  color: var(--color-accent-primary);

}
```

## P2-14 · Primary CTA 在全局尺寸不统一（56 vs 48）

**现象**（实测）
- Quiz bottom `[下一题]` primary: 343×56
- Result sticky `[进入 商]` primary: 约 343×56
- Dialog `[继续答题]` primary: 335×48
- 首页卡片（整卡可点）: 343×72

**问题本质**：min-height 在弹窗里 48，页内 sticky 56，不统一。

**修复方案**：`.btn` 全局 `min-block-size` 按 SC-TT1（44-56），弹窗保持 48，sticky CTA 保持 56（这实际上是分级规范，不是 bug）。建议写入 button-system-spec.md 明确：

```
- btn in dialog: min-height 48 (secondary)
- btn sticky CTA: min-height 56 (primary)
- btn inline: 44
```

然后 CSS 用 modifier 区分：`.btn--lg { min-block-size: 56px }`，避免各处用 class 组合时高度失控。

## P2-15 · Replay confirm 按钮顺序不符合破坏性操作规范

**现象**
replay_confirm 是 alertdialog（破坏性），当前顺序 `[ghost cancel autofocus, primary ok]`。autofocus 在 cancel 符合防误触，但 ghost cancel 在移动端仍然不够显眼。

**问题本质**：破坏性 alertdialog 的 cancel 应该是 outline 不是 ghost（防止用户无意中点了 primary）。

**修复方案**：app.js 815 行把 `variant: 'ghost'` 改成 `variant: 'outline'`：

```javascript
{ label: t('dynasty.replay_confirm.cancel'), variant: 'outline', autofocus: true, onClick: function () {} },
```

## P2-17 · dialog close 后 aria-hidden 未还原

**现象**
card-detail 关闭时，app.js 1599 行 `appRoot.removeAttribute('aria-hidden')` 已经在 closeDetail 里做了。但我在走查时进入 wrong 页 snapshot 几乎为空（只有 status live region 有内容），推测某个路径 rerender 时 aria-hidden 未清。

```javascript
// app.js 276-294 render 切换前
var appEl = document.getElementById('app');
if (appEl) appEl.removeAttribute('aria-hidden');
```

## P2-18 · 结算页 primary 箭头图标方向不随 RTL 翻转

**现象**（app.js 1367 行）
"进入 商 →" 把 arrow-right 硬编码在文案后。如果未来支持 RTL（阿语），箭头方向错。

**问题本质**：i18n 响应式未前置处理。

**修复方案**：用 CSS logical property 翻转：

```css
:dir(rtl) .btn svg[data-icon="arrow-right"] { transform: scaleX(-1); }
```
